# 可能的实现路径

这份文档列出讨论过的几条实现路径，从最容易到最根本排序。它们不是互斥的——可以先做容易的验证产品形态，同时探索根本的方向。

## 路径 A：在现有架构上"打补丁"

**思路**：把异步行为串行化进 token 流，用现成模型微调。

**做法**：
- 在词表里加入 `<gap:2s>`、`<gap:30s>` 这类时间间隔 token，把时间分桶（对数尺度，比如 8-12 个桶：<1s, 1-3s, 3-10s, 10s-30s, 30s-2m, 2-10m, 10m-1h, >1h）。
- 加入"沉默 token"，让模型每个时间片都决策"发言还是沉默"。
- 训练数据：Discord/Slack 公开服务器导出、IRC 日志、GitHub issue 评论流。这些数据有精确时间戳和消息边界。
- 微调一个 7B 左右的小模型（Qwen / Llama / Mistral），LoRA 即可。
- 推理用一个事件循环 wrapper：每秒检查模型预测的下一个 token 是 `<gap>` 还是实际内容；如果是 `<gap>` 就等待对应时长；期间用户发新消息就注入上下文重新决策。

**优点**：
- 不需要新架构，几个月就能 demo。
- 数据已存在，只是需要清洗。
- 工程量可控，单人可做。

**缺点**：
- **本质上是把异步串行化**——模型仍然是同步的，所谓"打断"只能在 token 边界检测，再中止生成、把新输入塞进上下文、重新发起推理。是用同步循环模拟异步。
- 天花板有限，是回合制模型的伪装。

**适合**：快速验证产品形态、建立初步评测体系。

## 路径 B：双向流式架构

**思路**：让架构本身就是双向流式的，模型在持续消费输入流的同时持续产生输出流。

**做法**：
- 离散化时间：每 100ms 一个 step（或者按字符到达时间 ground 时间步）。
- 双流输入：在每个 step 模型同时接收"我方上一个输出"和"对方上一个输入"，每个 token 都可以是"沉默"。
- Recurrent backbone：SSM 或 RWKV，维护一个状态向量。Transformer 太重，每步重算所有 attention 在物理时间步上跑不动。
- 双头输出：每个 step 输出两个分布——预测自己的下一个 token 和对方的下一个 token。训练时两个都算 loss，推理时只采样自己的。
- 状态可持久化：状态向量可以 dump，会话可以挂起、恢复、迁移。

这是 Moshi 在语音上做的事，移植到文本是可行但没人做过。

**优点**：
- 真双向并发，不是模拟。
- 状态显式，天然适合持久化、跨会话。
- 跟"持续存在的状态机"目标一致。

**缺点**：
- **没有 pretrained 模型可用**，几乎要从头训。
- 训练数据要"双流时间戳"格式，需要把消息按字符级展开，工程量大。
- 评测真空，benchmark 全部要重做。
- 是研究项目，1-2 年级别。

**适合**：长期目标，团队/机构投入。

## 路径 C：emit / draft / discard 机制（推荐）

**思路**：每步预测两个 token——一个 meta token 控制是否"提交"，一个 content token 是实际内容。meta 决定 content 是否进入对话流。

**做法**：

词表扩展，加入几个 meta token：
- `<emit>`：下一个内容 token 实际发送。
- `<draft>`：下一个内容 token 是草稿，不发送，但保留在内部。
- `<discard>`：丢弃所有未发送的草稿，重新开始内部生成。
- `<wait>`：本步不生成内容，等待。

训练数据格式示例：
```
<user>你觉得这个方案怎么样
<draft>我<draft>觉<draft>得<discard>
<draft>这<draft>个<draft>方<draft>案<emit>这<emit>个<emit>方<emit>案<emit>有<emit>意<emit>思
<wait><wait><wait>
<emit>但<emit>我<emit>有<emit>个<emit>问<emit>题
```

每个内容 token 前面有一个 meta token 决定它的命运。

训练目标：标准 next-token prediction。词表里有 meta token，模型同时学到"何时 emit"和"emit 什么"，被同一个 loss 优化。

KV cache 管理（关键）：
- **committed cache**：emit 过的 token，是真实对话历史。
- **draft cache**：未 emit 的 token，可能被 discard。
- discard 时回滚 draft cache。这跟 speculative decoding 的 rollback 机制几乎一样，可以直接复用现有基础设施。

**为什么这条路最有力量**：

1. **概念上对应人类真实行为**——思考、编辑、提交。说出口的话只是内部语言流里被"提交"的部分。
2. **架构上不需要发明新东西**——标准 Transformer 加几个 token，可以拿任何预训练模型做 LoRA 微调。
3. **工程上能借力**——和 speculative decoding 在结构上同构，KV cache 管理、回滚机制有现成代码。
4. **训练上有清晰目标**——next-token prediction，不需要新的 loss 设计。
5. **天花板高**——`discard` 操作给了模型"擦写"能力，配合无限运行循环，架构上跨越到图灵完备的形态。

**关键设计选择（待定）**：

- **draft 进不进 KV cache？** 进的话模型有"内心独白"——记得刚才想说什么，但 cache 会被未提交内容污染。不进的话每次 emit 重新开始，干净但失去连续性。倾向于"进 cache，但用不同 segment embedding 区分"。
- **meta 和 content 是串行还是并行预测？** 倾向于并行——同一个 forward pass 两个 head，meta 作为 gating。
- **何时学会 discard？** 这是个非平凡的信用分配问题。冷启动用合成数据模仿，进阶用 RL（reward 给 emit 序列质量）。

**起手版本**：emit / draft / discard 三个 token，先不加 wait（时间问题留到下一步）。Qwen2.5-7B 上做 LoRA 微调，合成几千条带 draft/discard 的对话数据，看模型能不能学会"想了想觉得不对就丢弃重来"。

## 架构底座：Transformer vs SSM

**Transformer + KV cache**：
- 短期可行的实现选择。
- KV cache 充当状态。
- 状态大小随历史增长，长期运行不经济。
- 现成生态成熟，能拿到任何预训练模型热启动。

**SSM（Mamba / RWKV / xLSTM）**：
- 长期更合适。
- 固定大小的隐状态向量，更"干净"地对应状态机。
- discard 时回滚状态更容易（直接恢复到 emit 时的状态向量）。
- 状态可以原生地在时间上延续，跟"持续存在的实体"目标一致。
- 状态可以 dump 到磁盘几小时后加载回来——成本恒定。
- 缺点：远距离精确召回弱、生态远不如 Transformer 成熟、scaling 还没完全验证。

**策略**：路径 C 在 Transformer 上验证概念；长期迁移到 SSM。

## 训练视角：双层优化框架

可以把整个问题建模为 bilevel optimization：

- **外层（慢变量 / 策略层）**：决定行为模式——发言、沉默、等待、听。决策稀疏，状态变化慢。
- **内层（快变量 / 内容层）**：在外层选定的模式下生成具体 token。

**为什么这个建模有用**：

1. 把"时机"和"内容"解耦——单层 next-token prediction 把所有决策压在一起，时机信号会被吞没。
2. 时间尺度分离——"现在该不该插话"和"说话时用哪个词"是不同频率的决策，物理上对的。
3. 数据问题被自然解决：
   - 外层数据：消息边界 + 时间戳。海量、容易拿。
   - 内层数据：消息内容本身。已有的文本预训练数据。
4. 对应可实现的架构：慢的 controller（低频，可以是小模型/SSM）+ 快的 generator（高频，现有 LLM）。

**注意**：bilevel 是优化框架，不直接是架构。可以用 bilevel 训练单一网络，也可以训练分层网络但不用 bilevel。这两件事要分开。

**实践上**：完全的 bilevel 训练（内层完全收敛后外层更新）在线对话里做不到。现实的做法是 two-timescale stochastic optimization——外层学习率比内层小一个量级，actor-critic 风格。这有成熟的收敛理论（Borkar）。

## 评测真空（共同的难题）

无论走哪条路径，评测都是从零设计的。需要至少包含：

- **时机判断**：给一段对话历史，模型选择发言还是沉默；人工标注 ground truth。
- **节奏匹配**：快节奏 IM 里回复延迟应该短，慢节奏邮件里应该长。
- **打断恢复**：被打断后能不能合理衔接。
- **多线程**：能不能在两个话题间切换。
- **draft 质量**（仅路径 C）：discard 的内容是否真的应该被丢弃；emit 的内容是否真的应该被发出。

没有评测就没法迭代。这是大多数 side project 死掉的原因。从一开始就要建。

## 起手建议

如果是从零开始：

1. **第一周**：在公开 IM 数据（Discord export 或 GitHub issue 时间戳序列）上跑出路径 A 的最小原型，确认端到端 pipeline 通畅。
2. **接下来一两个月**：转向路径 C，扩展词表加入 emit/draft/discard，合成训练数据，做 LoRA 微调。
3. **同时**：搭最初的评测集，哪怕只有几十个手标样本。
4. **长期**：把路径 C 的方案迁移到 SSM 底座；探索路径 B 的双向流式架构。

第一周后就知道这事 work 不 work。如果 work，剩下的都是工程和打磨。如果不 work，问题大概率出在数据格式或训练目标上，调整后再来一轮。
